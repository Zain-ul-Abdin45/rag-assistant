# RAG Assistant

A local retrieval-augmented generation app. Upload documents, ask questions, get streamed answers grounded in your own content — all running on your machine with no external API keys.

**Stack:** FastAPI · PostgreSQL + pgvector · Ollama · PyMuPDF · plain JavaScript frontend

## Architecture

### Data Flow

```
Browser → POST /documents
       → file saved to uploads/
       → background job: extract → chunk → embed → pgvector
       → job status polled via GET /jobs/{id}

Browser → POST /chat (includes session_id)
       → query embedded
       → recall_memory: scan vec_conversations for similar past Q+A pairs
       → search: scan vec_chunks, filter by relevance threshold (L2 < 1.0)
       → if no relevant chunks: return "off-topic" message, show no sources
       → build prompt: doc context + memory context + chat history
       → Ollama streams answer token by token
       → response streamed as NDJSON
       → save_turn: embed + persist Q+A to vec_conversations for future recall
```

### Backend Modules

| Module | Responsibility |
|---|---|
| `main.py` | HTTP routes, lifespan, background task dispatch |
| `db.py` | Schema creation for all 5 tables |
| `ingest.py` | PyMuPDF extraction, chunking, parallel Ollama embedding |
| `rag.py` | Vector search with relevance threshold, memory recall/save, Ollama streaming |
| `summarize.py` | Generates and persists a summary for a document |
| `config.py` | Single source of truth for all env vars |
| `logger.py` | File + console handler, respects `LOG_LEVEL` |

### Database Schema

```
documents        — id, filename, filepath, pages, created_at
chunks           — id, doc_id, chunk_text, chunk_index
vec_chunks       — chunk_id → chunks, embedding vector(768)

conversations    — id, session_id, turn_id, role, content, created_at
vec_conversations — conversation_id → conversations (user rows only), embedding vector(768)
```

`session_id` is generated by the browser (`crypto.randomUUID`) and persisted in `localStorage` — memory accumulates across page refreshes on the same device.

Retrieval uses pgvector L2 distance. Chunks with distance ≥ 1.0 are excluded — off-topic queries return no sources instead of misleading results.

## Quick Start (local, no Docker)

### Prerequisites

- Python 3.12+
- PostgreSQL with pgvector installed
- Ollama running locally

### 1. Create env.local

```env
DATABASE_URL=postgresql://postgres:your_password@localhost:5432/rag_assistant
OLLAMA_HOST=http://localhost:11434
EMBED_MODEL=nomic-embed-text
CHAT_MODEL=llama3.2
EMBED_DIM=768
UPLOAD_DIR=uploads
CHUNK_SIZE=800
CHUNK_OVERLAP=150
TOP_K=5
EMBED_WORKERS=3
LOG_LEVEL=DEBUG
```

### 2. Install dependencies

```bash
python -m venv .rag_assistant
source .rag_assistant/bin/activate
pip install -r requirements.txt
```

### 3. Pull Ollama models

```bash
ollama pull nomic-embed-text
ollama pull llama3.2
```

### 4. Start the app

```bash
uvicorn main:app --reload --host 127.0.0.1 --port 8000
```

Open `http://127.0.0.1:8000`.

```bash

docker compose exec ollama ollama pull nomic-embed-text
docker compose exec ollama ollama pull llama3.2
```

## Configuration Reference

All variables are loaded from `env.local` (local) or set as environment variables (Docker / Railway).

| Variable | Default | Description |
|---|---|---|
| `DATABASE_URL` | — | Full PostgreSQL connection string |
| `OLLAMA_HOST` | `http://localhost:11434` | Ollama server URL |
| `EMBED_MODEL` | `nomic-embed-text` | Embedding model |
| `CHAT_MODEL` | `llama3.2` | Chat and summarization model |
| `EMBED_DIM` | `768` | Vector dimension (must match model) |
| `UPLOAD_DIR` | `uploads` | Directory for uploaded files |
| `CHUNK_SIZE` | `800` | Characters per chunk |
| `CHUNK_OVERLAP` | `150` | Overlap between adjacent chunks |
| `TOP_K` | `5` | Chunks retrieved per query |
| `EMBED_WORKERS` | `3` | Parallel embedding threads (use `1` on CPU-only Ollama) |
| `LOG_LEVEL` | `INFO` | `DEBUG` · `INFO` · `WARNING` · `ERROR` |

## Logging

Log verbosity is controlled entirely by `LOG_LEVEL`.

| Level | What gets logged | Use when |
|---|---|---|
| `DEBUG` | Everything, including per-chunk embed progress | Local debugging |
| `INFO` | Startup, uploads, ingest start/done, chat queries | Local dev |
| `WARNING` | Only warnings and errors | Production |
| `ERROR` | Only hard errors | High-traffic production |

High-frequency polling endpoints (`GET /health`, `GET /jobs/{id}`, `GET /documents`) have had their debug lines removed — they do not spam logs even at `LOG_LEVEL=DEBUG`.

## What Needs Further Improvement

### Phase 1 — LlamaIndex migration (next)

The internal ingestion and retrieval code in `ingest.py` and `rag.py` is hand-rolled. Conversation memory is already implemented natively (pgvector + `conversations` table). Replacing the chunking and retrieval internals with LlamaIndex adds multi-format document loaders and better node-level metadata with no FastAPI route changes.

Target replacements:

| Current | LlamaIndex equivalent | Status |
|---|---|---|
| Manual chunking in `ingest.py` | `SimpleNodeParser` | Pending |
| Ollama HTTP calls for embedding | `OllamaEmbedding` | Pending |
| Raw psycopg pgvector insert/query | `PGVectorStore` + `VectorStoreIndex` | Pending |
| Manual context assembly in `rag.py` | `ContextChatEngine` | Pending |
| Session memory via `conversations` table | `ChatMemoryBuffer` | Already built |

The FastAPI routes and PostgreSQL schema stay unchanged. Only the internals of `ingest.py` and `rag.py` are swapped.

### Phase 2 — LangGraph orchestration

After LlamaIndex is in place, wrap it with a LangGraph state graph to handle multi-step workflows explicitly:

- Parallel ingestion branches (extract → embed → summarize as concurrent nodes)
- Conditional retrieval (decide whether to search docs, memory, or both)
- Structured agent behavior with observable state transitions

### Multi-format document support

Currently PDF-only. Planned loaders (most have direct LlamaIndex equivalents):

- DOCX (`python-docx`)
- TXT / Markdown
- HTML
- Images with OCR

### Extraction quality improvements

- Table extraction from PDFs
- Section-aware chunking (respect headings and structure)
- Better metadata: page numbers, section titles, document dates

### Auth & user isolation

No authentication layer exists. Planned:

- JWT-based login (FastAPI + python-jose)
- Per-user document namespacing in PostgreSQL
- Row-level security so users only see their own documents

### Caching

No caching layer. High-value targets:

- Embedding cache: identical text should not re-embed (Redis key: `embed:{sha256(text)}`)
- Query result cache: repeated questions return instantly
- Background job queue: replace FastAPI `BackgroundTasks` with RQ or Celery for reliability and retries

### Observability

No metrics or tracing. Planned additions:

- Structured JSON logging (swap current plaintext format)
- Prometheus metrics endpoint (`/metrics`) for ingest throughput, chat latency, embedding time
- OpenTelemetry tracing for end-to-end request visibility

### Markdown export

Documents should be exportable as `.md` after ingestion for archival, review, or downstream RAG indexing.
